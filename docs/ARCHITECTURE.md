# SingerOS 架构设计文档

> 基于 **Digital Assistant Platform** 理念构建的企业级 AI 操作系统
> 
> **版本：2.0** | **最后更新：2026-04-21**

## 1. 核心愿景

构建一个企业级数字员工平台，让企业可以像管理真实员工一样，创建、配置、授权、调度和审计 AI 数字员工。

数字员工不是单纯的聊天机器人。它需要有独立身份、接收任务的入口、真实执行工作的环境，以及模型、工具、技能、知识库等基础能力。

**设计原则：**
- **分层解耦** - 入口接入层、执行引擎层、基础能力层、业务使用层分离
- **绑定关系为核心** - IM身份、外部连接、工作节点构成数字员工的工作入口
- **数字员工是最高抽象** - 员工档案 + 绑定关系 + 能力配置 + 运行环境

**核心价值：**
- 让 AI 助手融入企业现有工作流程，而非要求员工学习新工具
- 通过标准化配置和管理，确保 AI 行为可控、可审计、可追溯
- 支持多场景、多渠道、多能力的灵活组合，满足企业多样化需求

## 2. 核心概念

### 2.1 数字员工组成

一个可工作的数字员工由四部分组成：

```
数字员工 = 员工档案 + 绑定关系 + 能力配置 + 运行环境
```

**员工档案：** 定义它是谁、属于哪个部门、负责什么岗位、职责范围是什么。

**绑定关系：** 解决用户和系统如何找到它、它在哪里执行工作的问题。

**能力配置：** 定义它能使用哪些模型、工具、技能和知识库。

**运行环境：** 定义它如何理解任务、拆解任务、调用能力、执行并返回结果。

### 2.2 绑定关系架构

绑定关系是数字员工能进入企业工作流的关键：

```
数字员工
    |
    +-- IM 身份
    |     用户可以在 IM 中像找同事一样找它
    |
    +-- 外部连接
    |     外部系统可以通过通知、回调等方式告诉它发生了什么事
    |
    +-- 工作节点
          它可以在受控环境中执行编码、命令、文件操作等真实工作
```

**IM 身份**用于绑定独立的即时通讯机器人身份。绑定后，该员工会在 IM 中以自己的名字和头像出现，用户可以直接与它对话。

**外部连接**解决的是"系统如何把事情交给数字员工"的问题。例如代码平台的通知、业务系统告警、工单系统回调等。

**工作节点**解决的是"数字员工在哪里真正干活"的问题。绑定工作节点后，数字员工可以在受控环境中执行命令、修改文件、运行测试、分析代码、生成结果。

### 2.3 平台能力分层

平台能力从下到上分为四层：

| 层级 | 核心组件 | 职责 |
|------|----------|------|
| **基础能力层** | AI模型、工作节点、Skills/Tools、知识库、权限、审计 | 提供基础支撑能力 |
| **执行引擎层** | 任务理解、任务拆解、会话管理、能力调用、沙箱调度 | 核心执行逻辑 |
| **入口接入层** | IM身份、外部连接、业务系统接入 | 多种接入方式 |
| **业务使用层** | 数字员工管理、工作台、任务管理、定时任务、多员工协作、技能市场 | 业务功能 |

## 4. 入口接入层

入口接入层负责将数字员工接入企业的现有工作流，包含三种绑定关系：IM身份、外部连接和工作节点。

### 4.1 IM 身份

**职责：** 将数字员工绑定到企业即时通讯工具中的机器人，用户可以在 IM 中像找同事一样找它。

**支持的渠道：**
- 企业微信：支持私聊、群聊、@消息、卡片消息
- 飞书：支持私聊、群聊、富媒体消息
- 钉钉：支持工作台集成、群机器人
- Slack：支持 App 集成、Slash Commands
- 其他企业 IM 平台

**核心能力：**
- **身份映射**：将 IM 用户映射为平台内部用户
- **消息权限校验**：验证发送者是否有权限调用数字员工
- **消息审计**：记录所有 IM 消息用于审计
- **多渠道路由**：根据消息来源路由到正确的数字员工

**典型状态：**
- 未绑定：无法在 IM 中直接找到该员工
- 已绑定：员工拥有独立 IM 身份，用户可直接对话
- 异常：机器人凭证失效、回调配置错误或渠道不可用

**连接器实现：**
每个 IM 渠道需要实现 Connector 接口，负责：
- 注册 Webhook 接收端点或长连接
- 验证请求签名和合法性
- 解析消息格式并转换为内部事件
- 数字员工响应回写到 IM

### 4.2 外部连接

**职责：** 接收外部系统推送的通知，解决"系统如何把事情交给数字员工"的问题。

**支持的外部系统：**
- **代码平台**：GitHub、GitLab、Gitee 等
- **事件通知**：Webhook、API 回调等
- **业务系统**：工单系统、审批系统、告警系统等
- **企业内部系统**：通过 API 接入的自有系统

**典型通知类型：**
- 代码平台：PR/MR 创建、PR/MR 更新、代码提交、Issue 创建、评论新增等
- 告警系统：监控告警触发、告警恢复等
- 工单系统：工单创建、工单状态变更、审批流变更等
- 自定义事件：外部系统通过 API 创建的任务

**核心能力：**
- **事件类型配置**：选择订阅哪些类型的事件
- **请求签名和密钥管理**：验证请求来源的合法性
- **内容映射**：将外部事件内容映射为平台内部任务
- **幂等处理**：避免重复处理同一事件
- **失败重试**：事件处理失败时自动重试
- **异步回调**：支持长时间任务的异步结果回写
- **触发指定员工**：按事件类型触发指定的数字员工

**连接器实现：**
每个外部系统需要实现 Connector 接口，负责：
- 注册 Webhook 接收端点
- 验证签名（如 HMAC-SHA256）
- 解析事件格式并标准化
- 发布到消息总线或直接触发数字员工

**已实现的连接器：**
- GitHub：支持 PR、Issue、评论、推送等事件，完成签名验证
- GitLab：规划中
- 自定义 Webhook：规划中

### 4.3 工作节点

**职责：** 绑定数字员工的远程工作环境，解决"数字员工在哪里真正干活"的问题。

**支持的节点类型：**
- **容器沙箱节点**：基于 Docker/Kubernetes 的隔离执行环境
- **远程执行节点**：通过 SSH 连接到远程服务器执行任务
- **后续可扩展**：其他类型的执行环境

**核心能力：**
- **节点健康检查**：定期检查节点是否可用
- **资源容量管理**：监控节点的 CPU、内存、磁盘等资源
- **工作目录管理**：管理任务执行的工作目录、仓库目录、缓存目录、结果目录
- **资源限制**：限制任务可使用的 CPU、内存、磁盘、网络、执行时长
- **执行审计**：记录命令执行日志、文件变更、退出码、执行产物
- **安全隔离**：确保任务在隔离环境中执行，防止影响宿主机

**典型状态：**
- 未绑定：只能做纯问答或轻量工具调用，无法执行代码类任务
- 已绑定：可以执行代码类、文件类、自动化类任务
- 运行中：节点正在处理任务
- 异常：节点不可用，任务需要暂停、迁移或失败重试

**关键要求：**
- 数字员工不能直接在宿主机执行任务
- 所有代码执行、文件操作、命令运行都应发生在受控沙箱中
- 沙箱需要记录执行轨迹，便于审计、回放和问题定位

### 4.4 绑定关系通用能力

所有类型的绑定关系都需要支持以下通用能力：

- **创建绑定**：将数字员工与外部系统建立连接
- **修改绑定**：更新绑定的配置参数
- **启用和停用**：临时启用或停用某个绑定
- **测试连接**：测试绑定是否正常工作
- **健康检查**：定期检查绑定的健康状态
- **变更审计**：记录绑定关系的变更历史
- **权限校验**：验证操作者是否有权限修改绑定

## 5. 执行引擎层

执行引擎层是数字员工的核心执行逻辑，负责任务理解、任务拆解、会话管理、能力调用和沙箱调度。

### 5.1 任务理解

**职责：** 解析用户意图，理解任务目标和约束条件。

**核心流程：**
- 接收标准化事件（来自IM消息、外部连接、定时任务等）
- 提取任务关键信息（目标、参数、上下文）
- 加载数字员工配置（角色配置、能力配置、权限配置）
- 构建任务执行上下文

### 5.2 任务拆解

**职责：** 将复杂任务分解为可执行的子任务序列。

**核心流程：**
- 分析任务复杂度
- 拆解为原子化的子任务
- 确定子任务执行顺序和依赖关系
- 为每个子任务分配合适的能力（模型/skill/tool）

### 5.3 会话管理

**职责：** 维护多轮对话上下文和任务状态。

**核心能力：**
- 短期记忆：会话级别的上下文维护
- 状态跟踪：记录任务执行进度
- 变量管理：临时变量和中间结果存储
- 会话恢复：支持中断后的恢复

### 5.4 能力调用

**职责：** 根据任务需求调用合适的AI模型、Skills或Tools。

**核心流程：**
- 根据任务类型选择AI模型
- 加载数字员工配置的可用Skills/Tools
- 构建提示词上下文
- 调用LLM执行推理
- 处理工具调用请求
- 聚合执行结果

### 5.5 沙箱调度

**职责：** 在受控工作节点中执行任务。

**核心流程：**
- 选择合适的工作节点
- 创建隔离沙箱环境
- 注入必要的环境变量和凭证
- 执行命令或脚本
- 收集执行结果和日志
- 清理沙箱环境

## 6. 基础能力层

### 6.1 AI 模型管理

**职责：** 统一管理企业可用的 AI 模型供应商、模型配置和调用策略。

**核心功能：**
- **模型供应商管理**：支持多家模型供应商和本地部署模型
- **模型目录**：维护模型名称、能力标签、上下文长度、成本、延迟、适用场景等信息
- **凭证管理**：管理 API 密钥、组织 ID、项目 ID、代理地址等凭证信息，支持密钥轮换
- **路由策略**：按数字员工、任务类型、成本预算、可用性选择合适的模型
- **降级策略**：主模型失败后切换备用模型
- **配额控制**：按租户、部门、数字员工、任务限制 token 用量和费用
- **调用审计**：记录模型、输入摘要、输出摘要、token 用量、耗时、错误和成本

### 6.2 工作节点管理

**职责：** 管理数字员工执行真实任务的基础设施。

**核心功能：**
- **镜像管理**：管理容器镜像
- **节点注册**：注册新的工作节点
- **节点健康检查**：定期检查节点是否可用
- **节点资源容量统计**：统计节点的资源使用情况
- **节点标签**：为节点打标签用于调度
- **可调度状态**：设置节点是否可接受新任务
- **沙箱生命周期管理**：管理沙箱的创建、启动、停止、销毁
- **工作目录管理**：管理任务执行的工作目录
- **网络策略**：配置节点的网络访问策略
- **凭证注入**：安全地注入执行所需的凭证
- **命令执行审计**：记录所有命令执行用于审计

### 6.3 Skills 能力系统

**Skill 定义：** 可复用的 AI 能力单元，是 SingerOS 的核心构建块。

**Skill 元数据：**
- 唯一标识符
- 名称和描述
- 版本号
- 分类（集成类、AI 类、工具类、工作流类）
- 输入输出模式定义
- 权限声明

**Skill 分类：**
- **集成类 Skills** - 外部系统集成（GitHub、GitLab、飞书等）
- **AI 类 Skills** - 基于大模型的推理能力（代码审查、摘要生成、分类等）
- **工具类 Skills** - 底层工具能力（Shell 执行、Python 脚本、HTTP 请求等）
- **工作流类 Skills** - 复杂编排能力（PR 审查工作流、Bug 分类工作流等）

**技能加载方式：**
- 文件系统：通过 SKILL.md 文件定义
- 代码嵌入：编译时打包的内置技能
- 远程加载：从技能市场动态下载（规划中）

### 6.4 Tools 工具系统

**Tool 定义：** 底层原子能力，提供与外部系统交互的具体实现。

**与 Skills 的区别：**
- Tools 是原子操作，Skills 可以组合多个 Tools
- Tools 由系统注册，Skills 可以由用户创建
- Tools 侧重执行，Skills 侧重智能决策

**内置 Tools：**
- HTTP 请求工具
- Shell 命令执行
- Python 脚本执行
- 文件读写操作
- 数据库查询工具

### 6.5 知识库

**职责：** 为数字员工提供企业内部上下文信息。

**核心功能：**
- **知识导入**：从文档、代码仓库、数据库等来源导入知识
- **知识分类**：按领域、项目、部门等维度分类知识
- **知识检索**：支持数字员工查询相关知识
- **权限控制**：控制不同数字员工和用户可访问的知识范围
- **知识更新**：支持知识的增量更新
- **使用统计**：统计知识的使用情况

**典型应用场景：**
- 代码仓库知识：帮助数字员工理解代码结构和规范
- 业务文档知识：帮助数字员工理解业务流程和规则
- 技术规范知识：帮助数字员工遵循企业的技术规范
- 历史案例知识：帮助数字员工参考历史解决方案

### 6.6 权限与审计

**权限管理：**
- **调用权限**：哪些用户可以调用该数字员工
- **知识库访问权限**：数字员工可以访问哪些知识库
- **工具执行权限**：数字员工可以执行哪些工具
- **网络访问权限**：数字员工是否允许访问外部网络
- **写入权限**：数字员工是否允许修改外部系统的数据

**审计能力：**
- **配置变更审计**：记录数字员工配置的变更历史
- **任务执行审计**：记录所有任务的执行过程和结果
- **工具调用审计**：记录所有工具的调用情况
- **模型调用审计**：记录所有模型的调用情况
- **执行回放**：支持回放任务的执行过程

## 7. 数字员工管理

### 7.1 员工档案

**基础信息：**
- 名称：数字员工的显示名称
- 头像：数字员工的头像图片
- 描述：数字员工的职责和能力说明
- 部门：数字员工所属的部门
- 岗位：数字员工的岗位角色
- 负责人：管理该数字员工的企业用户
- 启用状态：启用或停用该数字员工

**角色配置：**
- **职责范围**：定义数字员工负责的工作领域
- **工作边界**：定义数字员工不能做的事情
- **系统提示词**：定义数字员工的行为准则和回复风格
- **输出格式**：定义输出结果的格式要求
- **拒绝策略**：当遇到超出职责范围或违反边界的请求时的回应方式
- **需要人工审批的场景**：定义哪些操作需要人工审批后才能执行

### 7.2 绑定关系管理

**IM 身份管理：**
- 渠道名称
- 机器人名称和头像
- 绑定状态
- 最近消息时间
- 操作入口：绑定、解绑、测试连接

**外部连接管理：**
- 来源系统名称
- 已订阅的事件类型
- 连接健康状态（正常/异常）
- 最近触发时间
- 操作入口：配置、禁用、测试事件

**工作节点管理：**
- 节点名称
- 节点类型
- 运行状态
- 当前任务数
- 资源规格（CPU、内存、磁盘）
- 操作入口：管理节点、切换节点、查看日志

### 7.3 能力配置

**默认模型：** 数字员工默认使用的 AI 模型。

**可用工具：** 数字员工可以调用的外部工具。

**可用技能：** 数字员工可以使用的预定义技能。

**可用外部服务：** 数字员工可以调用的外部服务能力。

**可用知识库：** 数字员工可以查询的企业知识库。

**默认工作节点：** 数字员工执行任务时默认使用的工作环境。

### 7.4 权限配置

**调用权限：** 哪些用户可以调用该数字员工。可以按用户、部门、角色进行配置。

**知识库访问权限：** 数字员工可以访问哪些知识库。

**工具执行权限：** 数字员工可以执行哪些工具，特别是涉及写操作的工具。

**网络访问权限：** 数字员工是否允许访问外部网络。

**写入权限：** 数字员工是否允许修改外部系统的数据。

## 8. 事件流示例

### 8.1 GitHub PR 自动审查流程

1. **事件触发** - 开发者创建 PR，GitHub 发送 Webhook
2. **事件接收** - 外部连接（GitHub Connector）接收 Webhook 请求
3. **签名验证** - 验证 Webhook 签名确保来源合法
4. **事件标准化** - 将 GitHub 格式转换为内部标准事件
5. **事件发布** - 发布到 RabbitMQ 的 PR 事件队列
6. **事件订阅** - 执行引擎层订阅该队列并消费事件
7. **任务理解** - 解析 PR 事件，提取变更信息和审查要求
8. **路由匹配** - 根据事件类型和配置匹配到对应的数字员工
9. **配置加载** - 加载数字员工的完整配置（角色、能力、权限）
10. **任务拆解** - 将 PR 审查任务拆解为子任务（获取代码、分析变更、生成评论）
11. **工作节点准备** - 选择并准备工作节点，创建沙箱环境
12. **代码获取** - 在沙箱中克隆仓库并检出 PR 分支
13. **上下文构建** - 获取 PR 差异内容，构建提示词上下文
14. **技能注入** - 注入代码审查相关的 Skills
15. **工具注入** - 注入 GitHub API 相关的 Tools
16. **大模型推理** - 调用 LLM 分析代码变更并生成审查意见
17. **工具执行** - 调用 GitHub API 发布 Review 评论
18. **执行审计** - 记录命令执行日志、文件变更、退出码等
19. **结果记录** - 将执行结果持久化到事件表和任务表
20. **响应返回** - 完成整个处理流程

### 8.2 IM 消息处理流程

1. **消息接收** - 用户在企业微信/飞书中发送消息给数字员工
2. **身份验证** - IM 身份连接器验证消息来源
3. **用户映射** - 将 IM 用户映射为平台内部用户
4. **权限校验** - 验证用户是否有权限调用该数字员工
5. **事件转换** - 将 IM 消息转换为内部事件
6. **任务创建** - 根据消息内容创建任务
7. **任务执行** - 执行引擎层处理任务
8. **结果回写** - 将执行结果发送回 IM 渠道
9. **消息审计** - 记录完整的消息和响应日志

### 8.3 定时任务执行流程

1. **触发器激活** - Cron 调度器到达预定时间
2. **任务创建** - 根据定时任务配置创建任务实例
3. **员工选择** - 获取绑定的数字员工
4. **模板加载** - 加载任务模板和预设参数
5. **任务执行** - 执行引擎层处理任务
6. **结果记录** - 记录执行结果和日志
7. **通知发送** - 如有配置，发送执行结果通知
8. **失败重试** - 如执行失败，按配置进行重试

### 9.1 多租户隔离

- **数据隔离**：不同租户的数据完全隔离
- **资源隔离**：不同租户的资源独立分配

- **任务执行审计**：记录所有任务的执行过程和结果

- **敏感数据脱敏**：自动脱敏敏感数据

## 总结

**SingerOS 架构核心：**

事件驱动 + 技能组合 + 执行引擎 + 数字员工抽象

**架构特点：**
- 清晰的分层设计，职责明确（入口接入层、执行引擎层、基础能力层、业务使用层）
- 绑定关系为核心（IM身份、外部连接、工作节点）
- 事件驱动的松耦合架构
- 企业级的多租户和权限支持
- 可插拔的 Skills 和 Tools 系统
- 受控的执行环境（容器沙箱）

**演进路线：**
- 第一阶段（MVP）：数字员工基础、外部连接、工作节点、PR审查流程
- 第二阶段：IM身份绑定、多员工协作、定时任务、知识库集成
- 第三阶段：企业级能力，多租户和可观测性
- 第四阶段：生态系统，技能市场和开发者平台
